# 潮汐孤岛（CCGame）

一个基于 **Vanilla JavaScript + Vite** 的 2D 生存建造网页游戏。玩家会出生在一片程序生成的群岛中，白天采集资源、制作工具和建造设施，夜晚抵御从海岸出现的敌人，尽可能延长生存天数并提高分数。

## 项目特点

- **动森风格 UI**：参考 [animal-island-ui](https://github.com/guokaigdg/animal-island-ui) 设计语言（薄荷青绿主色、温暖大地色文字、奶油米白背景、pill 圆角与游戏按键 3D 立体阴影、Nunito + Noto Sans SC + Zen Maru Gothic 字体）重构界面。
- **桌面 & 移动端通用**：自动识别触屏环境，提供虚拟摇杆、攻击/交互/冲刺按钮、左右面板抽屉、背包长按菜单等触屏交互。桌面端支持 `M`/`B`/`C` 快捷键随时显示或隐藏海图、背包、制作面板。
- **轻量音效**：通过 Web Audio API 现场合成短音，无需额外资源；覆盖采集、攻击、击杀、建造、烹饪、取水、进食、钓鱼、受伤、UI 交互等关键操作。
- **程序化群岛地图**：使用噪声与岛屿描述符生成大世界，并按区块动态加载。
- **生存循环完整**：包含生命、饥饿、口渴、体力、昼夜变化与天气提示。
- **采集 / 制作 / 建造**：可采集木材、石块、纤维、浆果、椰子等资源，制作石斧、石镐、长矛、鱼竿、篝火、木墙、木地板、雨水收集器、混合种子包和种植箱。
- **战斗与防守**：夜晚会出现“暗潮爬行者”，需要通过武器和营地布局进行防守。
- **交互式 UI**：包含背包、制作面板、海图、世界目标面板、右键物品菜单和提示文本。
- **钓鱼系统**：可对近处水面抛竿，在咬钩窗口内收线获得不同鱼类。
- **种植 / 烹饪**：可在种植箱中培育南瓜，并在篝火旁把鱼获烤成熟食。

## 当前玩法

1. 开始游戏后先在岛上采集树木、岩石、灌木和棕榈。
2. 通过右侧制作面板合成基础工具与建造套件。
3. 左键切换当前手持物，靠近目标后进行采集、攻击、建造或使用。
4. 使用篝火补充夜间营地收益，利用木墙、地板和种植箱搭建营地。
5. 建造雨水收集器补充饮水，或通过鱼竿在浅水 / 深水区域捕鱼并烹饪成烤鱼。
6. 把浆果和纤维制成混合种子包，在种植箱中培育南瓜补充食物。
7. 在夜晚击退敌人，尽可能维持角色状态并提高生存评分。

## 操作说明

### 桌面端

- `W / A / S / D`：移动
- `Shift`：冲刺
- `鼠标左键`：行动 / 攻击 / 使用当前手持
- `鼠标右键（场景）`：取消当前手持
- `E`：与当前目标交互
- `M`：显示 / 隐藏 海图面板
- `B`：显示 / 隐藏 背包面板
- `C`：显示 / 隐藏 制作面板
- `背包左键`：切换当前手持物品
- `背包右键`：打开物品菜单
- `Esc`：关闭物品菜单

### 移动端 / 触屏

- **左下虚拟摇杆**：移动角色（支持斜向）
- **红色按钮**：攻击 / 使用当前手持（按住可连续触发）
- **青色按钮**：与当前目标交互
- **黄色按钮**：按住冲刺
- **顶部圆形按钮**：分别切换“状态/海图”与“背包/制作”面板
- **轻点画布**：设置瞄准方向并触发一次主动作
- **拖动画布**：调整瞄准方向（不会触发动作）
- **背包格长按**：打开物品菜单

## 本地运行

### 环境要求

- Node.js 20.x (与仓库当前 GitHub Pages 工作流保持一致)
- npm

### 安装依赖

```bash
npm ci
```

### 启动开发环境

```bash
npm run dev
```

### 构建生产版本

```bash
npm run build
```

构建产物输出到 `dist/` 目录。

## 联机功能（实验性）

基于 **WebRTC + Trystero** 实现的零后端联机：默认使用公共 BitTorrent tracker 作为 WebRTC 信令通道，不需要自建服务器。如果公共 tracker 不稳定 / 你处于严格 NAT 或企业内网导致 P2P 无法建立，可在联机弹窗右上角的 ⚙ 设置里切换到自建中转：

- **自定义 WebSocket 信令中转**：使用 `@trystero-p2p/ws-relay`，只转发 WebRTC 握手，数据仍走 P2P。适合公共 tracker 不通但 P2P 本身没问题的网络。
- **完整数据中转**：使用本仓库 [`server/`](./server) 下自建的 WebSocket 服务器，**所有游戏数据均通过服务器转发**，完全不依赖 WebRTC。适合 P2P 根本不通的网络（对称 NAT、UDP 全屏蔽等）。

### 使用方法

1. 在开始界面点击右下角「联机」按钮打开房间面板。
2. 填写昵称（仅在本地保存）。
3. **创建房间**：点击「创建房间」会自动生成 6 位房间码。把房间码分享给好友。
4. **加入房间**：在输入框中粘贴房间码，点击「加入房间」。
5. 进入房间后可在面板内文字聊天、查看成员列表、显示与各 peer 的实时延迟。
6. 点击「离开房间」断开连接。

### 自定义中转服务器

点击联机弹窗右上角 ⚙ 按钮可打开「自定义中转服务器」设置，三种模式可选：

- **公共 BitTorrent tracker**（默认）：无需任何服务器，但在部分网络下可能不稳定。
- **自定义 WebSocket 信令中转**：使用 `@trystero-p2p/ws-relay` 协议，仅承担 WebRTC 信令转发（**不会承载游戏数据**）；P2P 建立后数据仍然端到端加密直传。适合公共 tracker 被屏蔽但 P2P 本身没问题的网络。
- **完整数据中转**：使用本仓库 [`server/`](./server) 下自建的 WebSocket 服务器，**所有游戏数据均通过此服务器转发**。完全不依赖 WebRTC，是 P2P 完全不通的网络（对称 NAT、UDP 全屏蔽、严格企业内网）下唯一能稳定工作的方案。

设置保存在浏览器本地（localStorage），下次自动读取使用。**同一房间内所有玩家必须使用相同的中转服务器地址**才能互相发现，请把地址连同房间码一起分享给好友。

### 自建中转服务器

下面分别说明两种自建方案。如果只是公共 tracker 不稳定，选「信令中转」即可；如果连 P2P 也根本建立不起来，请选「完整数据中转」。

#### 方案 A：自建 WebSocket 信令中转（数据仍走 P2P）

`@trystero-p2p/ws-relay` 提供了一份非常小的 WebSocket 信令服务器，**只负责帮助玩家互相发现，不存储任何游戏数据**，几行代码就能运行起来。

##### 1. 准备一台公网服务器

任何能跑 Node.js 18+ 的 VPS / 云主机即可。如需通过 `wss://`（强烈推荐）访问，需要绑定一个域名并配置 TLS（自建可用 Caddy / Nginx 反代，也可以由 Cloudflare 之类的服务商终结 TLS 再回源到 `ws://`）。

##### 2. 安装并启动 relay

新建一个空目录，初始化并安装：

```bash
mkdir ccgame-relay && cd ccgame-relay
npm init -y
npm install @trystero-p2p/ws-relay
```

新建 `server.js`：

```js
import { createWsRelayServer } from '@trystero-p2p/ws-relay/server';

createWsRelayServer({ port: 8080 });
console.log('CCGame relay listening on ws://0.0.0.0:8080');
```

`package.json` 中加上 `"type": "module"`，然后启动：

```bash
node server.js
```

建议通过 `systemd` / `pm2` / `docker` 等方式保活：

```bash
# 使用 pm2 简单跑起来
npm install -g pm2
pm2 start server.js --name ccgame-relay
pm2 save
```

##### 3. 配置 TLS（推荐）

浏览器在 `https://` 页面下只允许连接 `wss://`，所以生产环境一定要走 TLS。

使用 Caddy 作为反向代理是最快的方案（自动申请 Let's Encrypt 证书）。`Caddyfile` 示例：

```caddy
relay.example.com {
  reverse_proxy 127.0.0.1:8080
}
```

`caddy run` 后即可通过 `wss://relay.example.com` 连接。

##### 4. 在游戏里启用

1. 打开联机弹窗，点击右上角 ⚙。
2. 选择「自定义 WebSocket 信令中转」。
3. 在文本框中填入服务器地址（每行一条，可填多个做冗余）：

   ```text
   wss://relay.example.com
   ```

4. 点击「保存并应用」。
5. 把"中转服务器地址 + 房间码"一起分享给好友——所有人必须设置相同的服务器，才能在同一个房间里互相发现。

##### 5. 防火墙 / 端口

- 自建在 8080：放行入站 TCP/8080（或经由 Caddy 反代时只需放行 80/443）。
- 信令中转服务器只在 WebRTC 握手阶段被使用，建立连接后玩家之间直接 P2P 通讯，所以即使你的 relay 带宽不大也完全够用。

#### 方案 B：自建完整数据中转（P2P 不通时使用）

当方案 A 也无法让玩家建立 P2P 连接时（例如对称 NAT、UDP 被防火墙整体封禁、企业内网仅放行 wss:443），请使用本仓库自带的完整数据中转。它不依赖 WebRTC，所有游戏消息（INPUT / SNAPSHOT / ENTITY_DELTA / INVENTORY / 聊天等）通过 WebSocket 直接经服务器透传。

启动：

```bash
cd server
npm install
npm start            # 默认监听 0.0.0.0:8090
# PORT=9000 HOST=127.0.0.1 npm start  # 自定义端口 / 监听地址
```

TLS 反代示例与方案 A 相同（仅需把端口改成 8090）。详见 [`server/README.md`](./server/README.md)。

在游戏里启用：

1. 打开联机弹窗 → ⚙ → 选择「完整数据中转」。
2. 填入服务器地址（每行一条，可多个做故障转移）：

   ```text
   wss://relay.example.com
   ```

3. 保存并应用，把"中转服务器地址 + 房间码"分享给好友。

> ⚠️ 完整中转模式下服务器需要承担所有玩家间的数据流量。4 人房间稳态约 30–80 KB/s，8 人满员高峰约 0.5–1 MB/s，请根据房间规模评估带宽。

### 网络说明

- 默认使用公共 STUN（Google / Cloudflare）。绝大多数家庭网络可直接 P2P 互联。
- 在严格 NAT 环境下 WebRTC 可能无法穿透；如果连 STUN/ws-relay 都救不了，请切换到 **完整数据中转** 模式（见上文「自建中转服务器 · 方案 B」），所有数据将走 WebSocket 经服务器转发。
- 默认信令依赖公共 BitTorrent tracker，受限网络可能屏蔽；遇到大厅扫不到房间 / 进房后看不见房主等情况，请按上文「自建中转服务器」搭建一台中转。

## 项目结构

```text
CCGame/
├─ index.html              # 页面结构与 HUD / 弹层
├─ styles/                 # 全局样式
├─ src/
│  ├─ app/                 # 启动、输入绑定、主循环、移动端控件
│  ├─ components/          # 地块、资源、敌人、结构定义
│  ├─ core/                # 常量、状态、DOM 引用
│  ├─ data/                # 物品与配方数据
│  ├─ ecs/                 # 简单 ECS 核心
│  ├─ inventory/           # 背包与物品操作
│  ├─ render/              # 场景绘制与特效
│  ├─ systems/             # 行为、运行时与工厂系统
│  ├─ ui/                  # 面板、菜单、提示交互
│  ├─ world/               # 地图生成与世界辅助逻辑
│  ├─ net/                 # 联机：Trystero 传输、协议、会话
│  └─ workers/             # 区块生成 Worker
├─ vite.config.mjs         # Vite 配置
└─ .github/workflows/      # GitHub Pages 部署流程
```

## 技术说明

- 使用原生 Canvas 进行渲染。
- 通过模块拆分组织 ECS、渲染、UI、世界生成与物品系统。
- 使用 Web Worker 处理世界区块生成，降低主线程压力。
- 使用 Vite 进行开发与构建，`base: './'` 方便静态部署。
